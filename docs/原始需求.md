# 1.目标
本系统采用“多入口扫描、统一证据层、分问题分析”的设计。  
- 多入口扫描：从大客户、应用场景、芯片、竞对四个入口采集招聘信号。

- 统一证据层：将招聘信息沉淀为“公司/主体—岗位—标签—历史特征”。

- 分问题分析：由不同 Agent 针对不同业务问题输出结论与动作建议。

举例说明：
- 大客户：老客户是否出现新项目/新方向机会？

- 场景拓新：目标场景是否出现可进入的新客户机会？

- 芯片机会：是否暴露我方代理线匹配需求或替代窗口？

- 竞对预警：竞对/原厂是否在组织扩张或渠道策略变化？
# 2.扫描对象

对于招聘信息的监控基于4个对象：

1. 大客户监控  

2. 场景拓新  

3. 芯片机会识别（品牌、品类、技术栈）  

4. 竞对预警

这一条很关键，但我建议你不要命名成“流失预警”，而是拆成两个方向：

### A. 供应链切换信号

判定规则（满足任一即触发）：

① 采购岗JD中出现"供应商开发""供应商导入""新供应商"

② JD中出现"国产替代""降本""成本优化" + 结合芯片关键词判断是否涉及我方代理线

③ JD中出现竞品芯片型号，而该客户原来用我方芯片 例：原来用STM32的客户，JD出现GD32/APM32

④ 招聘"元器件分析师""成本工程师"类岗位

⑤ 大量招聘采购+同期研发岗位减少（业务收缩+供应链重组）

信号强度：

商机含义 & 销售动作：

⚠️ 这是防守型信号！必须快速响应

👉 FAE紧急确认：客户是否在评估替代方案

👉 如果是降本需求：主动提供性价比更优的替代型号 （例：STM32→STM32C0，用我方代理线内的低成本方案守住）

👉 如果是国产替代政策要求：推我方代理的国产线

👉 升级客户关系层级，必要时申请特殊价格支持

关键词：  
JD 出现

- 国产替代
    
- 供应商开发
    
- 新供应商导入
    
- 二供开发
    
- 成本优化
    
- 电子元器件采购开发
    
- 关键器件资源导入
    
- 替代方案评估

## 数据整理
### 衍生字段表  
  
> 说明：这些字段无法从招聘完整上直接获取，需要基于每次抓取的原始数据，通过规则逻辑自动生成或更新。  
  
| 字段名                    | 中文名称    | 说明                 | 公式                                                                                                                                                                                          |     |
| ---------------------- | ------- | ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --- |
| `crawled_at`           | 抓取时间    | 本次实际执行抓取的时间戳       | 每次抓取时自动记录当前系统时间                                                                                                                                                                             |     |
| `source_url_canonical` | 规范化链接   | 去除查询参数后的标准化职位详情页链接 | 取职位详情页的完整URL，截取 ? 之前的部分作为标准链接。<br>例：https://xxx.com/job/123.shtml?a=1 → https://xxx.com/job/123.shtml                                                                                       |     |
| `source_job_id`        | 平台岗位ID  | 该职位在招聘平台上的唯一标识符    | 从 source_url_canonical 中提取：取最后一个 / 与文件后缀名（.shtml 或 .html）之间的数字部分。<br>例：https://xxx.com/job/123456.shtml → 123456                                                                            |     |
| `first_seen_at`        | 首次发现时间  | 该职位第一次被抓取到的时间      | 当某个 source_job_id 首次出现在抓取结果中时，将当次的 crawled_at 写入此字段。<br>此字段一旦写入，永不更新                                                                                                                        |     |
| `last_seen_at`         | 最近发现时间  | 该职位最近一次被抓取到的时间     | 每次抓取中，若该 source_job_id 出现在结果中，则用当次的 crawled_at 覆盖更新此字段。                                                                                                                                     |     |
| `times_seen`           | 累计抓取次数  | 该职位被成功抓取到的总次数      | 初始值为 1（首次发现时）。<br>之后每次抓取结果中出现该 source_job_id，则 times_seen = times_seen + 1。                                                                                                                 |     |
| consecutive_misses     | 连续未出现次数 | 该职位连续未出现在抓取结果中的次数  | 初始值 = 0。<br>每次抓取后：<br>• （`source_job_id`）若本次出现 → 重置为 0<br>• 若本次未出现 → +1                                                                                                                     |     |
| `disappeared_at`       | 确认下架时间  | 判定该职位已下架/失效的时间     | 当 consecutive_misses >= 3 时，写入当次 crawled_at。<br>若职位重新出现，则清空此字段。。                                                                                                                            |     |
| `record_status`        | 岗位记录状态  | 标记该职位当前所处的生命周期阶段   | 按以下优先级判断（从上到下依次匹配）：<br>① disappeared_at 有值 → 填写 CLOSED<br>② job_description_text 与上次记录不一致 → 填写 JD_CHANGED<br>③ 该 source_job_id 是首次出现 → 填写 NEW<br>④ 该 source_job_id 已有历史记录且无上述变化 → 填写 ACTIVE |     |

**字段更新时序图**

本次抓取执行
  │
  ├─ 记录 crawled_at = 当前时间
  │
  ├─ 对抓取到的每条职位（出现在结果中）：
  │    ├─ source_url_canonical、source_job_id（照常生成）
  │    │
  │    ├─ 若首次出现：
  │    │    ├─ first_seen_at     = crawled_at
  │    │    ├─ times_seen        = 1
  │    │    ├─ consecutive_misses = 0
  │    │    └─ record_status     = NEW
  │    │
  │    └─ 若已存在：
  │         ├─ last_seen_at      = crawled_at
  │         ├─ times_seen       += 1
  │         ├─ consecutive_misses = 0        ← 重置
  │         ├─ disappeared_at    = 清空       ← 如之前有值
  │         ├─ JD 有变 → record_status = JD_CHANGED
  │         └─ JD 无变 → record_status = ACTIVE
  │
  └─ 对本次【未出现】在结果中的历史职位：
       ├─ consecutive_misses += 1
       │
       ├─ 若 consecutive_misses >= 3：
       │    ├─ disappeared_at  = 本次 crawled_at
       │    └─ record_status   = CLOSED
       │
       └─ 若 consecutive_misses < 3：
            └─ 不做变更，等待下次抓取继续观察

### 时序特征计算

该公司本月（最近30天） vs 上月（最近60天到最近30天）招聘发布量变化（环比）  
#### 对公司招聘变化

指标：
- **新增岗位数（最近30天）**, 用于看：最近有没有新动作。
-定义：                            
最近30天新增岗位数 = first_seen_at 在本月的岗位数                
比变化 = (本次最近30天 - 最近60天到最近30天) / 最近60天到最近30天 × 100%                        

| 指标       | 含有                                     |
| -------- | -------------------------------------- |
| ① 最近30天  | first_seen_at 在最近30天的岗位数== 客户新放了多少坑位      |
| ② 本月关闭数  | disappeared_at 在最近30天的岗位数 = 有多少岗位招到人了/不招了 |
| ③ 当前在招总数 | status = ACTIVE 的岗位数 = 此刻公司还在招多少人      |

| 场景       | 新增  | 关闭  | 解读           |
| -------- | --- | --- | ------------ |
| 新增↑ 关闭→  | 暴增  | 正常  | 🟡 扩产/新项目    |
| 新增→ 关闭↑  | 正常  | 暴增  | ⚫ 裁员/缩编/项目砍掉 |
| 新增↑ 关闭↑  | 都多  | 都多  | 人员流动大，可能重组   |
| 新增↓ 关闭→  | 减少  | 正常  | 招聘放缓，观望      |
| 新增→ 关闭=0 | 正常  | 没有  | 一直招不到人，岗位都挂着 |
