# Product Requirements Document - job_scrape

**Author:** Hans
**Date:** 2026-03-12

## Executive Summary

job_scrape 是面向展芯科技市场、销售与 FAE 团队的内部招聘情报平台。MVP 阶段优先解决“可信采集与证据留存”，目标是用稳定、低人工干预的采集流程替代手工复制、粘贴和计算，为后续业务级 AI 分析提供可信输入。

当前痛点不止是人工耗时，更是数据质量、可追溯性和可复用性不足。手工表格流程会导致采集不稳定、证据链断裂、历史状态不可连续对比，进而削弱信号识别、趋势分析和业务判断可靠性。在大客户监控场景中，底层不稳定会持续放大后续分析噪音。

MVP 的核心价值是先建立可信证据层与采集层：在 51job、猎聘等平台执行低频、定向、可追溯采集，完成标准化存储、状态跟踪和基础导出。该阶段不是“降级目标”，而是产品路线中的必要约束；只有先做到“采得稳、可验证、可累计”，后续 AI 判断才具备业务价值。

从长期定位看，job_scrape 将演进为内部 B2B 情报平台；从当前阶段看，它首先是一套可信采集基础设施。本 PRD 以该分阶段定位为核心，确保 MVP 成功标准、范围边界和演进路径聚焦于当下可验证目标。

### What Makes This Special

job_scrape 的差异化不在于“先上 AI”，而在于先解决 AI 依赖的可信采集问题。相较手工表格流程，它显著减少复制、粘贴和手工计算，同时提升运行稳定性；相较一次性爬虫脚本，它强调证据留存、状态追踪与持续可维护性；相较通用招聘监控工具，它服务的是销售情报场景，而非 HR 运营场景。

产品核心洞察是：招聘数据应被视为长期沉淀的数据资产，而非一次性抓取结果。若证据层和采集层不稳定，下游 AI 只会放大噪音，无法形成可信、可追溯、可复用的业务情报。因此，MVP 优先级是“更可靠地获取与管理数据”，而不是“更复杂地解释数据”。

## Success Criteria

### User Success

- 登录并手动触发后，系统自动完成采集，用户无需复制、粘贴或手工计算
- 每次运行结束后，用户可以直接取用结构化结果，无需二次加工
- 每周人工操作时间从当前约 2-3 小时降低至 15 分钟以内（主要为登录与触发操作）
- 用户能通过导出结果核验采集完整性，不依赖记忆或手工对账

### Business Success

- 3 个月内：证明主链路可替代人工采集流程；TOP10 客户在两个平台的覆盖率达到 90% 以上；任务执行成功率 ≥ 85%（7 天滚动均值）
- 3 个月内：每周实际人工投入降至 15 分钟以内，自动化覆盖率达到 100%（登录/触发除外）
- 12 个月视角：在可信采集底座基础上，逐步叠加 AI 信号判断与销售/FAE 反馈闭环，使系统从采集工具演进为内部业务情报系统

### Technical Success

- 51job 主链路稳定：搜索 → 列表 → 详情采集完整跑通，7 天滚动成功率 ≥ 85%
- 猎聘 PoC 可复现：至少 1 条稳定查询链路，能在 7 天内重复执行并产出有效样本
- 结构化存储与去重：统一字段输出，job_fingerprint 生成，last_seen_at 自动更新，重复岗位不累加
- 岗位状态跟踪：first_seen_at / last_seen_at / disappear_count 自动维护，支持 active / suspected_removed 状态切换
- 简化版证据留存：每条岗位保留 source_url、采集时间戳、原始 HTML 文件路径，关键字段缺失时自动告警
- 重试机制：失败任务自动进行指数退避重试（最少 3 次），超限后状态置为 manual 待人工处理
- 基础导出：支持 CSV/Excel/JSON，可按客户、平台、时间范围筛选

### Measurable Outcomes

| 指标 | MVP 目标 |
|------|----------|
| 人工干预次数（登录触发除外）| 0 |
| 每周人工操作时长 | ≤ 15 分钟 |
| TOP10 客户平台覆盖率 | ≥ 90% |
| 7 天滚动任务成功率 | ≥ 85% |
| 衍生字段抽样误差 | < 5% |
| 证据可追溯比例（source_url + HTML 路径）| 100% |

## Product Scope

### MVP - Minimum Viable Product

- **51job 稳定采集**：搜索、分页、列表、详情全链路，包含重试逻辑与失败状态管理
- **猎聘最小 PoC**：最小可用采集链路验证，记录 challenge/captcha 事件
- **结构化存储与去重**：统一字段写入、job_fingerprint 生成、重复合并
- **岗位状态跟踪**：first_seen_at / last_seen_at / disappear_count / record_status 自动维护
- **简化版证据留存**：source_url + 采集时间戳 + HTML 文件路径，支持原文回查
- **重试与人工兜底**：指数退避重试（最少 3 次），超限置为 manual
- **基础导出**：CSV/Excel/JSON，支持按客户、平台、时间范围筛选

### Growth Features（Post-MVP）

- AI 信号分析层（双层 Prompt：侦察兵 + 质检员）
- 情报卡自动生成与钉钉/飞书推送
- 销售/FAE 情报卡反馈标注（已跟进 / 误报 / 暂不处理）
- 定时自动调度（n8n 或 Python cron）
- 截图证据留存与完整风控指标面板
- BOSS直聘、智联招聘、脉脉平台扩展

### Vision（未来）

一套完整的内部销售情报业务系统：招聘数据自动采集 → AI 双层信号判断 → 情报卡自动推送 → 销售/FAE 反馈闭环 → 市场部追踪情报转化率。系统随使用时间积累客户行为基线，AI 判断越来越准，销售与 FAE 可以基于系统输出直接决定跟进优先级，而不是依赖人工感知。

## User Journeys

### Journey 1：小陈 — 标准采集成功路径

**背景**：小陈，展芯科技市场部情报运营员，她最清楚每周手工流程的代价——打开四五个标签页、逐个搜索客户名、复制粘贴、手工计算新增和关闭数，再发给 GPT 生成分析，最后整理发给销售。周一早上，她开始新的一周。

**第一幕：触发不再是繁琐的开始**
小陈打开系统，用账号登录 51job 和猎聘的已保存会话。上周系统已经跑过一次，所以这次是增量更新。她选择"执行 TOP10 客户采集"并触发，系统开始工作。她去倒了杯咖啡。

**转折点：等待，而不是操作**
过去这个时间段是最费力的：搜索、复制、整理。现在她只是等着进度条走。系统自动翻页、进入详情、写入数据库、计算衍生字段。中间有一条任务遇到慢响应，系统自动退避等待后重试成功，小陈完全不知道。

**高潮：拿到可用的结构化结果**
15 分钟后，任务完成。小陈打开覆盖率报告：TOP10 客户 9/10 覆盖（一家猎聘采集失败，状态 manual），本次新增岗位 23 条，3 条岗位状态从 active 变为 suspected_removed。她导出 Excel，字段已经全部计算好，first_seen_at / disappear_count 都在，不需要手工填。

**结局：今天的工作不超过 15 分钟**
她把导出文件发给销售群，并在群里备注了 3 条岗位变化。这是过去需要花 2-3 小时才能完成的工作。唯一剩下的任务：处理那一条 manual 状态的任务。

---

### Journey 2：小陈 — 采集中断与故障恢复路径

**背景**：同一天，但针对猎聘采集链路，系统碰到了反爬挑战。

**第一幕：出现问题，但系统没有崩溃**
任务执行中，猎聘返回了一个验证码挑战。系统记录了 captcha_hit 事件，自动暂停并等待后重试。第一次重试成功率低，进入第二次退避。第三次重试后，任务仍然失败，状态自动变为 `manual`，系统不再继续阻塞其他任务。

**关键时刻：小陈收到需要干预的信号**
系统将该任务标记到 manual 队列，小陈在系统中看到通知。她查看风控事件记录：3 次 captcha_hit，平台返回了挑战页，session 可能已被标记。

**小陈的判断**：今天猎聘窗口期不好，不强行跑。她选择"明天手动重放这条任务"，并将其余 manual 任务确认为"今日跳过"。这个决定被记录在任务日志里。

**结局：有损降级，而不是全局失败**
51job 这边 9 条客户数据全部成功。猎聘 1 条 manual，明天继续。整个系统没有崩溃，小陈知道发生了什么，也知道怎么处理。这和过去完全没有记录的手工失败完全不同。

---

### Journey 3：小陈 — 配置管理路径

**背景**：销售昨天告诉小陈，华为一个子品牌被纳入新的重点监控客户名单，要加进 TOP10 白名单。

**第一幕：不需要找技术人员**
小陈打开配置中心，找到"公司监控白名单"。她直接添加新公司名称和所需平台（51job + 猎聘），设置优先级，保存。不需要改代码，不需要等技术人员部署。

**关键时刻：配置立刻生效**
下一次采集触发时，系统自动将新公司纳入任务队列。first_seen_at 从这次采集开始计时，历史状态从零积累。

**结局：情报基线从第一次采集就开始建立**
三周后，小陈能看到这家公司的岗位变化趋势。这是系统设计里最安静也最重要的价值：数据资产随使用时间自动积累，不需要额外操作。

---

### Journey 4：小陈 — 数据导出与人工分析衔接路径

**背景**：MVP 阶段还没有 AI 分析层，但小陈需要把系统输出的结构化数据提供给销售做判断。这是 Growth 阶段 AI 推送的前置手工版本。

**第一幕：导出是工作流的出口**
本周采集结束后，小陈选择按"本周新增 + 状态变化"筛选，导出 Excel。字段已包含 job_title、company、location、salary、publish_time、first_seen_at、disappear_count，不需要手工计算。

**关键时刻：可交付的数据，而不是原始截图**
她把导出文件发给负责华为账户的销售小王，并备注："本周新增 5 条技术岗，2 条消失，详见文件第 3 行。"小王不需要自己去平台查，直接看表格做判断。

**结局：情报从系统流向销售，手工但有据可查**
这不是最终形态（最终应该是自动推送情报卡），但比之前"小陈手工抄写 + 口头汇报"更可靠、更可追溯。这条旅程本身就是 Growth 阶段价值的前哨：当 AI 推送就绪时，这个步骤将被自动化取代。

## B2B Internal Platform Specific Requirements

### Project-Type Overview

job_scrape 是单租户内部工具，不涉及多租户架构、付费订阅分层或面向外部客户的账号体系。部署目标是展芯科技内部网络或私有环境，用户规模极小（1-5 人），架构设计应以简单、可维护为优先，避免引入不必要的企业级复杂度。

### Permission Model

MVP 阶段采用两级权限模型：

| 操作 | 运营员（小陈）| 管理员（Hans / 技术人员）|
|------|-------------|----------------------|
| 触发采集任务 | ✓ | ✓ |
| 查看任务状态与日志 | ✓ | ✓ |
| 处理 manual 队列 | ✓ | ✓ |
| 修改公司监控白名单 | ✓ | ✓ |
| 修改采集参数与阈值 | ✓ | ✓ |
| 导出数据 | ✓ | ✓ |
| 查看风控指标面板 | ✓ | ✓ |
| 系统配置（环境变量、部署参数）| ✗ | ✓ |

MVP 阶段两个角色的功能权限基本一致，差异仅在底层系统配置层面。不需要实现复杂的 RBAC，简单的角色标记即可满足需求。

### Integration Requirements

**MVP 集成（必须）**

- **51job 平台**：Playwright 驱动的浏览器自动化采集，登录态持久化，搜索/列表/详情全链路
- **猎聘平台**：最小 PoC 链路，验证基础采集可行性，记录反爬事件
- **文件导出**：CSV / Excel / JSON，支持按客户、平台、时间范围筛选，字段完整输出

**Growth 集成（Post-MVP）**

- 钉钉 / 飞书表格写入（情报卡推送阶段）
- 其他招聘平台（BOSS直聘、智联、脉脉）
- 内部 CRM 或数据仓库对接（视后续需求确认）

### Data Retention & Compliance

**留存策略（当前默认）**

- 原始岗位数据默认永久保留，作为历史趋势分析的基础数据资产
- 采集证据（HTML 文件路径、source_url、时间戳）与岗位记录同步保留
- 风控事件日志至少保留 90 天，用于故障回溯和策略调整

**待明确事项（后续版本处理）**

- 正式确认数据留存周期与删除机制
- 明确岗位中包含的个人信息字段（如招聘联系人、薪资范围）的处理边界
- 建立最小化字段清单，避免采集超出业务必要范围的个人信息

**访问控制**

- 凭据（平台账号、代理配置）加密存储，不在日志明文输出
- 数据访问限于内部授权人员
- 操作日志支持按 task_id / session_id 追溯，满足基础审计需求

### Technical Architecture Considerations

**部署模型**

- 单实例部署，内部私有环境或本地服务器
- 不需要横向扩展架构（MVP 并发 ≤ 2）
- 配置通过环境变量或配置文件管理，无需配置管理服务

**会话与身份管理**

- 平台登录会话持久化（存储 cookies/session），减少重复登录干预
- 会话过期自动检测，触发人工重新登录提示
- 多平台账号独立管理，不共享 session 上下文

### Implementation Considerations

- 模块化采集器设计：每个平台独立 collector，互不耦合，新平台接入只需实现 collector + parser + 配置
- 错误分级处理：网络超时 / 反爬挑战 / 结构解析失败 分开处理，避免混用重试策略
- 本地存储优先（PostgreSQL 单实例），不引入分布式存储或消息队列（MVP 阶段）

## Project Scoping & Phased Development

### MVP Strategy & Philosophy

**MVP 方式**：问题解决型 MVP（Problem-Solving MVP）——先验证“可信采集”核心假设，而非构建完整平台。  
**核心问题**：能否用自动化稳定替代人工复制粘贴、手工计算的招聘情报采集流程？  
**资源需求**：1 名技术人员（全栈/后端）+ 1 名运营员（小陈）参与验收测试。  
**验证周期**：4 周 MVP 迭代。

### MVP Feature Set（Phase 1）

Phase 1 的能力边界与交付范围以 `## Product Scope` 中的 MVP 条目为准，不重复展开。该阶段的验收重点为：
- 支撑 Journey 1-4 的核心流程完整可用
- 满足 Success Criteria 的自动化率、覆盖率、成功率与人工时长目标
- 建立“失败可恢复、结果可追溯、配置可调整”的基础运营能力

### Post-MVP Features

**Phase 2（Growth）:**
- AI 双层信号分析（侦察兵 + 质检员）
- 情报卡自动生成与钉钉/飞书推送
- 销售/FAE 反馈标注与闭环追踪
- 定时自动调度（n8n / cron）
- 完整风控指标面板与截图证据留存

**Phase 3（Expansion）:**
- BOSS直聘、智联、脉脉平台接入
- 管理层汇总视图（情报跟进率、信号准确率趋势）
- 场景拓新 / 芯片机会识别 / 竞对预警
- 内部 CRM 或数据仓库对接

### Risk Mitigation Strategy

**Technical Risks:**
- 猎聘链路稳定性风险较高，MVP 成功标准以 51job 主链路为主，猎聘按 PoC 验证。

**Market Risks:**
- MVP 未包含情报卡与自动推送，需管理预期并强调“先验证可信采集”的阶段目标。

**Resource Risks:**
- 维护依赖少量人员，采用模块化与配置外置降低后续维护门槛。

## Functional Requirements

以下能力条目直接映射到已确认的用户旅程与成功指标：
- Journey 1/2/3/4 对应任务执行、故障恢复、配置管理与导出能力
- Success Criteria 对应自动化覆盖率、可追溯性与稳定性相关能力
- Product Scope（MVP）定义 FR1-FR29 的优先实现范围，FR30-FR33 为 Growth 能力

### 任务与执行管理

- FR1: 运营员可以手动触发指定客户范围的采集任务。
- FR2: 运营员可以查看任务执行状态（pending/running/success/failed/manual）。
- FR3: 运营员可以查看单任务执行日志与失败原因。
- FR4: 运营员可以对 manual 任务执行重放或跳过处理。
- FR5: 系统支持为每次任务生成可追溯的 task_id 与执行记录。

### 平台采集能力

- FR6: 系统支持按客户名称在 51job 执行岗位搜索与列表采集。
- FR7: 系统支持对 51job 列表结果执行详情采集并提取岗位核心字段。
- FR8: 系统支持在猎聘执行最小可用采集链路（搜索→列表→详情）。
- FR9: 系统支持记录采集过程中平台挑战事件（如 captcha/challenge）。
- FR10: 系统支持在采集失败后按策略重试并在超限后转入 manual。

### 数据标准化与状态跟踪

- FR11: 系统支持将多平台岗位数据标准化为统一字段结构。
- FR12: 系统支持为岗位生成唯一指纹用于去重与合并更新。
- FR13: 系统支持维护岗位的 first_seen_at 与 last_seen_at。
- FR14: 系统支持维护岗位 disappear_count 并支持状态变更标记。
- FR15: 系统支持保留岗位历史状态变化用于趋势回查。

### 证据与审计能力

- FR16: 系统支持为每条岗位记录保留 source_url。
- FR17: 系统支持为每条岗位记录保留采集时间戳。
- FR18: 系统支持为每条岗位记录关联原始 HTML 证据路径。
- FR19: 系统支持在关键字段缺失时生成可追溯告警记录。
- FR20: 系统支持按 task_id/session_id 定位采集证据与日志。

### 配置与运营控制

- FR21: 运营员可以维护公司监控白名单而无需改代码。
- FR22: 运营员可以调整采集参数与阈值（如频率、重试相关配置）。
- FR23: 系统支持在配置变更后将新配置应用到后续任务。
- FR24: 管理员可以维护系统级配置（环境变量、部署参数、凭据）。

### 结果消费与导出

- FR25: 运营员可以按客户、平台、时间范围筛选采集结果。
- FR26: 运营员可以导出采集结果为 CSV。
- FR27: 运营员可以导出采集结果为 Excel。
- FR28: 运营员可以导出采集结果为 JSON。
- FR29: 系统支持在导出结果中提供用于人工分析的完整核心字段集。

### Growth 阶段能力（后续）

- FR30: 系统支持基于岗位变化生成业务信号候选。
- FR31: 系统支持生成并推送情报卡到协作渠道（钉钉/飞书）。
- FR32: 销售与 FAE 可以对情报卡进行反馈标注。
- FR33: 系统支持追踪情报从发出到跟进的状态闭环。

## Non-Functional Requirements

### Performance

- NFR1: 在默认配置（TOP10 客户、2 平台、并发 ≤ 2）下，单次任务执行总时长 P95 ≤ 7 分钟；以任务运行日志按周统计。
- NFR2: 任务触发、状态查询、日志查看等关键操作接口响应时间 P95 ≤ 2 秒；以应用监控按日统计。
- NFR3: 在并发 ≤ 2 的运行约束下，任务丢失率应为 0%，重复执行率 ≤ 0.5%；以 task_id 去重审计按周统计。

### Reliability

- NFR4: 任务失败后系统应自动执行 3 次指数退避重试（1min/3min/9min），重试耗尽后自动转为 `manual`；以任务状态流转日志校验。
- NFR5: 单平台或单客户任务失败不应中断同批次其他任务，批次级中断率应为 0%；以批次运行日志按周统计。
- NFR6: 7 天滚动任务成功率（success / total）应 ≥ 85%；当连续 2 天低于阈值时触发告警。

### Security

- NFR7: 平台账号、代理配置与系统凭据必须加密存储，日志中敏感字段明文暴露率应为 0%；以每周日志抽样审计（≥200 条）验证。
- NFR8: 访问控制至少区分“运营员（任务与参数）”和“管理员（系统配置）”，未授权操作拦截率应为 100%；以权限测试用例验证。
- NFR9: 控制台与服务间数据传输应全部使用 TLS 通道，非加密连接占比应为 0%；以网络抓包抽检与网关日志验证。

### Data Governance & Compliance

- NFR10: 仅采集白名单字段，非白名单字段落库率应为 0%；以每日入库字段审计任务验证。
- NFR11: 原始岗位数据默认永久保留；同时系统需支持按 30/90/180 天策略切换归档或删除，策略变更在 24 小时内生效并可审计。
- NFR12: 岗位记录证据完整率（source_url + 采集时间戳 + HTML 路径）应 ≥ 99.5%；以每日数据质量报表验证。
- NFR13: 操作与采集日志需支持 `task_id/session_id` 双键追溯，指定任务 5 分钟内可定位完整链路；以故障演练记录验证。

### Maintainability

- NFR14: 平台采集器需模块化，新增平台改动不应触及现有平台核心逻辑文件（允许公共组件）；以代码变更范围审计验证。
- NFR15: 字段解析规则变更应可独立发布，回归测试通过率应 ≥ 95%，且不影响其他平台主流程；以回归报告验证。
- NFR16: 白名单、阈值、重试策略应配置化管理，硬编码配置项占比应为 0%；以配置扫描脚本按版本校验。

### Integration

- NFR17: CSV/Excel/JSON 导出字段一致性应为 100%，编码统一为 UTF-8；以每次发布后的三格式对比测试验证。
- NFR18: 相同筛选条件在 24 小时内重复导出，记录主键重复率应为 0%，导出失败重试成功率应 ≥ 95%；以导出任务日志验证。
- NFR19: Growth 阶段外部集成应通过可替换适配层接入，适配层变更不应影响核心采集流程回归通过率（≥95%）；以模块化回归测试验证。
