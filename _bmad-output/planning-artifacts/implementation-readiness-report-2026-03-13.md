## PRD 分析

### 功能需求（FR）

**任务与执行管理**
- FR1: 运营员可以手动触发指定客户范围的采集任务。
- FR2: 运营员可以查看任务执行状态（pending/running/success/failed/manual）。
- FR3: 运营员可以查看单任务执行日志与失败原因。
- FR4: 运营员可以对 manual 任务执行重放或跳过处理。
- FR5: 系统支持为每次任务生成可追溯的 task_id 与执行记录。

**平台采集能力**
- FR6: 系统支持按客户名称在 51job 执行岗位搜索与列表采集。
- FR7: 系统支持对 51job 列表结果执行详情采集并提取岗位核心字段。
- FR8: 系统支持在猎聘执行最小可用采集链路（搜索→列表→详情）。
- FR9: 系统支持记录采集过程中平台挑战事件（如 captcha/challenge）。
- FR10: 系统支持在采集失败后按策略重试并在超限后转入 manual。

**数据标准化与状态跟踪**
- FR11: 系统支持将多平台岗位数据标准化为统一字段结构。
- FR12: 系统支持为岗位生成唯一指纹用于去重与合并更新。
- FR13: 系统支持维护岗位的 first_seen_at 与 last_seen_at。
- FR14: 系统支持维护岗位 disappear_count 并支持状态变更标记。
- FR15: 系统支持保留岗位历史状态变化用于趋势回查。

**证据与审计能力**
- FR16: 系统支持为每条岗位记录保留 source_url。
- FR17: 系统支持为每条岗位记录保留采集时间戳。
- FR18: 系统支持为每条岗位记录关联原始 HTML 证据路径。
- FR19: 系统支持在关键字段缺失时生成可追溯告警记录。
- FR20: 系统支持按 task_id/session_id 定位采集证据与日志。

**配置与运营控制**
- FR21: 运营员可以维护公司监控白名单而无需改代码。
- FR22: 运营员可以调整采集参数与阈值（如频率、重试相关配置）。
- FR23: 系统支持在配置变更后将新配置应用到后续任务。
- FR24: 管理员可以维护系统级配置（环境变量、部署参数、凭据）。

**结果消费与导出**
- FR25: 运营员可以按客户、平台、时间范围筛选采集结果。
- FR26: 运营员可以导出采集结果为 CSV。
- FR27: 运营员可以导出采集结果为 Excel。
- FR28: 运营员可以导出采集结果为 JSON。
- FR29: 系统支持在导出结果中提供用于人工分析的完整核心字段集。

**Growth 阶段能力**
- FR30: 系统支持基于岗位变化生成业务信号候选。
- FR31: 系统支持生成并推送情报卡到协作渠道（钉钉/飞书）。
- FR32: 销售与 FAE 可以对情报卡进行反馈标注。
- FR33: 系统支持追踪情报从发出到跟进的状态闭环。

**FR 合计：33 条**

说明：FR1-FR29 为 MVP 范围；FR30-FR33 为 PRD 中已定义但明确归属 Growth 阶段的后续能力。

### Additional Requirements

**业务与范围约束**
- MVP 的核心假设是先验证“可信采集与证据留存”，而不是上线完整 AI 情报分析闭环。
- 项目为单租户内部工具，目标用户规模 1-5 人，应优先保持实现简单、可维护。
- 猎聘链路按 PoC 验证处理，MVP 成功标准以 51job 主链路稳定性为主。

**集成与部署约束**
- MVP 必须集成 51job、猎聘最小链路和三种导出格式；钉钉/飞书、CRM/数仓属于 Post-MVP。
- 部署模型为单实例内部私有环境或本地服务器，不引入分布式存储或消息队列。
- 配置通过环境变量或配置文件管理，无需独立配置管理服务。

**权限与数据治理约束**
- MVP 阶段采用两级权限模型：运营员与管理员。
- 凭据需加密存储，日志不得输出敏感信息明文。
- 默认永久保留原始岗位数据，同时要求系统支持 30/90/180 天策略切换。

#### 🟡 次要关注点（4 项）

**Issue 2：NFR 追溯在 Story AC 中基本具备，但文档级映射仍不完整**

- **现状**：Story 1.6、1.7、5.2 已分别显式承接 NFR6、NFR8、NFR7/NFR9，质量比旧版本明显提高。

- **建议**：补一小节“NFR Coverage Map”，把 NFR6/7/8/9/17 等直接映射到对应 Story，减少实现偏差。

**Issue 3：Story 1.2 缺少无效输入错误路径**

- **问题**：当前 AC 覆盖了成功路径，但没有明确“客户范围为空/非法/超范围”时的系统行为。
- **风险**：开发实现容易把输入校验下沉为隐式异常，导致 API 错误码和审计行为不一致。
- **建议**：补充一条 AC，要求对无效参数返回明确 `VAL_*` 错误码并写入失败原因。

**Issue 4：Story 2.1 缺少空结果与分页边界说明**

- **问题**：Story 2.1 已明确重试框架适用于 51job，这是正确修复；但仍未写清“搜索无结果”与“翻页结束条件”的期望行为。
- **风险**：开发者可能将“0 结果”错误处理化，或在分页终止条件上实现不一致。
