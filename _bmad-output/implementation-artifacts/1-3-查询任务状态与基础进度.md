# Story 1.3: 查询任务状态与基础进度

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 运营员,
I want 查看任务列表和单任务状态,
so that 我能实时掌握执行进展。

## Acceptance Criteria

1. Given 系统中存在已创建任务，When 运营员查看任务列表或详情，Then 系统返回 `pending/running/success/failed/manual` 状态。
2. 支持按时间范围与客户筛选，并按最近更新时间倒序返回。
3. 列表与详情接口遵循统一响应包装 `{data, meta, error}` 与统一错误码约定。
4. 本故事聚焦“查询可观测性”，不提前实现任务执行器、状态推进、日志详情与 manual 动作。
5. `GET /api/tasks` 的 `meta` 结构固定包含 `total`、`limit`、`offset`；`GET /api/tasks/{task_id}` 的 `meta` 固定为 `null`。
6. 时间筛选语义固定为：优先使用 `updated_at`，若记录无 `updated_at` 则回退 `created_at`；当 `start_time > end_time` 返回 `VAL_INVALID_TIME_RANGE`。

## Dev Notes

### Story Foundation

- 来自 Epic 1 的明确目标：在已具备任务创建（Story 1.2）的前提下，补齐“查得到、看得懂”的任务状态可视能力。
- 本故事是 Story 1.4（日志与失败原因）和 Story 1.5（manual 处理）的前置能力：先完成列表/详情状态查询，再扩展深度日志与动作闭环。
- 当前仓库中任务创建能力已存在，状态以 `pending` 起步并记录 `task.created` 事件；本故事重点是读取与展示。

### Technical Requirements

- 继续复用现有路由聚合方式：在 `app/api/routes/` 下新增查询端点，不引入 `api/v1` 重构。
- 统一时间语义：输入输出使用 UTC ISO8601；若 `start_time > end_time`，返回 `VAL_INVALID_TIME_RANGE`。
- 查询响应建议最小字段：
  - 列表项：`task_id`、`status`、`customer_scope`、`triggered_by`、`created_at`、`updated_at`
  - 详情：在列表字段基础上增加 `progress`
- `progress` 最小实现：
  - `current_status`: 与任务状态一致
  - `updated_at`: 任务最近更新时间
  - `event_count`: `task_events` 对应 `task_id` 的事件总数
- 排序规则：默认 `updated_at DESC, created_at DESC`，确保“最近变化优先可见”。
- `meta` 协议固定：
  - 列表：`meta={"total":<int>,"limit":<int>,"offset":<int>}`
  - 详情：`meta=null`

### Architecture Compliance

- 状态枚举必须与架构状态机一致，不新增自定义状态。
- 错误响应统一：
  - `data: null`
  - `meta: null`
  - `error: {code, message, details}`
- 查询接口不包含任何状态变更副作用；读写职责分离。
- 继续满足可追溯性：返回体至少包含 `task_id`，详情中的进度聚合来自 `task_events`。

### Library & Framework Requirements

- 保持现有技术栈：FastAPI `0.135.1`、SQLAlchemy `2.0.48`、Pydantic `2.12.5`、Alembic `1.18.4`、Pytest `8.4.2`、Ruff `0.13.0`。
- 不引入 Redis、消息队列、异步任务框架；本故事为查询面能力。

  - `job_scrape/app/api/routes/tasks.py`（新增 GET 端点）
