# Story 1.2: 手动触发采集任务并生成追溯ID

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 运营员,
I want 手动触发指定客户范围的采集任务,
so that 我可以发起可追溯的采集流程。

## Acceptance Criteria

1. Given 运营员已登录且提交有效客户范围，When 点击触发任务，Then 系统创建任务并返回唯一 `task_id`。
2. 初始状态固定为 `pending`，并持久化创建时间与触发人信息。
3. 任务创建遵循统一 API 包装 `{data, meta, error}`，并具备明确错误码风格。
4. 本故事仅覆盖“任务创建入口 + 初始审计记录”；不提前实现任务执行、状态推进、日志详情、manual 处理。

## Dev Notes

### Story Foundation

- 本故事是 Epic 1 的第一条业务故事，目标是建立“任务入口事实源”，不是执行采集本身。
- 该故事要为后续 Story 1.3/1.4/1.5 提供稳定主键与状态起点，因此最关键产物是：可创建的任务记录、唯一 `task_id`、可追溯审计事件。
- 当前范围只解决“创建”动作，不实现 worker、运行时状态流转、失败恢复、日志详情或 manual 队列。

### Technical Requirements

- `task_id` 必须由服务端生成，推荐 UUID4 字符串；返回给调用方并持久化。
- 初始状态必须固定为 `pending`，不能允许客户端覆盖。
- 需要保存触发人信息；由于 Story 1.7 才实现完整权限体系，本故事应引入“可替换的操作者上下文依赖”，而不是提前做完整登录系统。
- 客户范围建议使用结构化列表字段表达，最少支持“公司名称列表”场景；命名保持 `snake_case`。
- 所有任务创建都必须同步写一条审计事件，事件名使用 `task.created`。

### Architecture Compliance

- 严格遵守架构中的任务状态机起点：`pending -> running -> success|failed|manual`；本故事只负责进入 `pending`。
- 统一 API 返回格式：`{data, meta, error}`，不要返回裸对象或裸数组。
- 错误码保持架构前缀约定：验证类使用 `VAL_*`，内部错误使用 `INT_*`。
- 所有日志或审计记录至少包含 `task_id` 或后续可扩展的 `session_id`；本故事优先保证 `task_id`。
- 数据库演进必须通过 Alembic；禁止手工建表。

### Library & Framework Requirements

- 沿用现有技术栈：FastAPI、SQLAlchemy 2.x、Alembic、Pydantic 2.x、Pytest、Ruff。
- 不新增消息队列、Redis、Celery、后台调度框架；这些超出 Story 1.2 范围。
- 若需要时间字段，统一使用 UTC 感知时间或项目内一致策略，避免 naive datetime 混用。

### File Structure Requirements

- 基于当前仓库实际骨架做最小增量，不要为了对齐未来理想结构而大规模重构。
- 当前 API 聚合入口在 `app/api/routes/api.py`，本故事应在该结构下接入任务路由。
- 建议新增的最小文件集合：
  - `app/api/routes/tasks.py`
  - `app/api/schemas/task.py` 或同等最小 schema 文件
  - `app/services/task_service.py`
  - `app/models/task.py` 与 `app/models/task_event.py`，或按当前 `app/db` 组织做最小落点
  - `tests/` 下对应 API/服务测试文件
- 若为减少改动选择不立即引入完整 `api/v1` / `repositories` 分层，需保持命名清晰，并为后续重构留出边界。

### Testing Requirements

- 至少覆盖以下测试场景：
  - 成功创建任务时返回 201 或项目约定成功码，并包含唯一 `task_id`
  - 返回体中的 `status` 为 `pending`
  - 触发人和创建时间已持久化
  - 同步创建 `task.created` 审计事件
  - 非法客户范围请求被拒绝，并返回统一错误结构
- 测试优先使用现有 FastAPI `TestClient` 模式，避免引入额外测试基础设施。
- 不要移除或弱化现有 `tests/test_health_smoke.py`；它是 Story 1.1 的回归哨兵。
