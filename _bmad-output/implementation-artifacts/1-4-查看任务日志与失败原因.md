# Story 1.4: 查看任务日志与失败原因

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 运营员,
I want 查看任务执行日志和失败原因,
so that 我能快速定位问题并判断是否需要人工介入。

## Acceptance Criteria

1. Given 任务已进入执行阶段，When 运营员打开任务日志，Then 系统按时间顺序展示关键事件与错误信息。
2. 每条日志至少包含 `task_id` 或 `session_id`（推荐双键）以支持追溯。
3. 日志查询接口遵循统一响应包装 `{data, meta, error}` 与统一错误码约定。
4. 本故事聚焦“日志可观测性”，不提前实现 manual 动作（replay/skip）与成功率告警能力。
5. 当任务不存在时返回 `VAL_TASK_NOT_FOUND`；当分页参数非法时返回 422。

## Dev Notes

### Story Foundation

- 来自 Epic 1 的目标：在“任务可查”（Story 1.3）基础上，补齐“问题可定位”的日志视图能力。
- 本故事是 Story 1.5（manual replay/skip）的直接前置：先让运营员看清失败原因，再执行人工动作。
- 本故事仅提供日志读取与展示，不改变任务状态机流转。

### Technical Requirements

- 推荐端点：`GET /api/tasks/{task_id}/events`
- 排序语义固定：`created_at` 升序（旧→新），保障故障链路阅读体验。
- 分页语义：
  - `limit`: 默认 50，范围 1~200
  - `offset`: 默认 0，最小 0
- `meta` 契约固定：`{"total": <int>, "limit": <int>, "offset": <int>}`
- 事件输出最小结构建议：
  - `task_id`: string
  - `session_id`: string | null
  - `event_type`: string
  - `operator`: string
  - `payload`: object
  - `error_code`: string | null
  - `error_message`: string | null
  - `created_at`: ISO8601 UTC

### Architecture Compliance

- 严格遵循统一响应包装：成功 `{data, meta, error:null}`，失败 `{data:null, meta:null, error:{...}}`。
- 严格遵循错误码前缀规范（`VAL_*` / `NET_*` / `RISK_*` / `INT_*`）。
- 保持读写分离：本接口为只读查询，不得触发状态变更或补写事件。
- 日志与事件必须保留追溯键：至少包含 `task_id` 或 `session_id`。

### Library & Framework Requirements

- 沿用现有栈与版本，不做升级：FastAPI `0.135.1`、SQLAlchemy `2.0.48`、Pydantic `2.12.5`、Pytest `8.4.2`、Ruff `0.13.0`。
- 不引入消息队列、全文检索或日志平台 SDK；MVP 保持最小实现。

### File Structure Requirements

- 预计最小改动集合：
  - `job_scrape/app/api/routes/tasks.py`（新增 events 读取端点）
  - `job_scrape/app/api/schemas/task.py`（新增事件项与分页响应 schema）
